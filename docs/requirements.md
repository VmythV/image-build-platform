# 镜像构建平台需求文档

## 1. 项目目标

建设一个可私有化部署的镜像构建平台，提供前端管理界面和后端构建服务，支持用户通过 Dockerfile 或表单方式定义镜像构建内容，将构建任务分发到本机或远程构建主机执行，并将构建产物推送到指定镜像仓库。

平台最终需要支持两种交付形态：

1. 二进制包部署：下载可执行文件及必要依赖后直接运行。
2. Docker 容器部署：通过容器运行平台服务，并能按配置访问宿主机 Docker 或远程构建主机。

## 2. 用户角色

### 2.1 平台管理员

- 管理构建主机、镜像仓库、系统配置、用户和权限。
- 查看所有构建任务、构建日志、镜像版本和审计记录。
- 配置平台部署参数、并发限制、日志保留策略和凭据加密策略。

### 2.2 镜像维护者

- 创建和维护镜像项目。
- 编辑 Dockerfile、表单化构建配置、版本说明和分支说明。
- 发起构建、查看构建日志、推送构建产物。

### 2.3 只读查看者

- 查看镜像项目、版本图、构建记录和构建产物。
- 不允许修改主机、仓库、凭据或发起构建任务。

## 3. 核心模块

### 3.1 前端管理界面

前端需要提供完整后台管理能力，至少包含：

- 仪表盘：展示构建任务状态、主机状态、仓库状态、近期失败任务、平台资源概览。
- 构建主机管理：新增、编辑、删除、测试连接、查看架构和 Docker 能力。
- 镜像仓库管理：新增、编辑、删除、测试登录、设置拉取和推送用途。
- 镜像项目管理：管理镜像项目、版本、分支、Dockerfile、说明文档和构建历史。
- 构建任务管理：创建构建、查看队列、实时日志、成功结果、失败原因和重试入口。
- 镜像产物管理：查看已构建镜像、标签、摘要、目标仓库、构建来源和推送状态。
- 系统帮助：指导用户如何添加主机、查看主机架构、配置 Docker 访问和配置镜像仓库。
- 系统设置：配置并发数、默认仓库、日志保留、构建超时、代理、通知等参数。

### 3.2 后端服务

后端服务负责 API、权限、任务调度、构建执行、日志采集和数据持久化：

- 提供前端所需的 REST API 或 GraphQL API。
- 管理构建主机连接信息和镜像仓库凭据。
- 管理镜像项目、版本节点、分支关系和 Dockerfile 内容。
- 创建构建任务并根据架构、主机状态、标签和负载选择构建主机。
- 在目标主机执行 Docker 构建命令，实时采集 stdout/stderr 日志。
- 构建完成后按配置推送镜像到指定仓库。
- 记录构建任务的状态流转、耗时、错误原因、产物摘要和审计信息。

### 3.3 构建执行器

构建执行器是后端中直接操作 Docker 的部分，可以作为同一进程模块，也可以后续拆成 Agent：

- 支持本机 Docker socket 构建。
- 支持通过 SSH 连接远程主机并执行构建。
- 预留远程 Agent 模式，用于更安全、更稳定地管理大规模构建节点。
- 支持 amd64、arm64 等架构标识，并根据构建目标选择合适主机。
- 支持普通 Docker build，建议优先兼容 BuildKit。
- 后续可扩展 buildx 多架构构建。

## 4. 功能需求

### 4.1 构建主机管理

构建主机用于执行镜像构建任务。

#### 4.1.1 默认本机构建

- 平台默认提供本机构建主机。
- 二进制部署时，本机构建主机指运行后端服务的物理机或虚拟机。
- 容器部署时，本机构建主机通常指容器访问到的宿主机 Docker。
- 容器部署需要支持挂载宿主机 Docker socket，例如 `/var/run/docker.sock`。
- 系统需要明确提示：挂载 Docker socket 等同于给容器较高宿主机控制权限，生产环境需谨慎配置。

#### 4.1.2 远程构建主机

平台应支持添加远程构建主机：

- SSH 主机：通过 SSH 登录目标主机并执行 Docker 命令。
- Docker API 主机：通过远程 Docker API 连接构建主机，作为可选增强。
- Agent 主机：后续可部署轻量 Agent 主动注册到平台。

主机配置字段建议包括：

- 主机名称。
- 连接方式：本机 Docker、SSH、Docker API、Agent。
- 主机地址、端口、用户名。
- SSH 密钥或密码，推荐优先支持密钥。
- Docker socket 路径或 Docker API 地址。
- 主机架构：amd64、arm64、arm/v7 等。
- 操作系统信息。
- Docker 版本、BuildKit 支持情况、buildx 支持情况。
- 主机标签：如 `linux`、`gpu`、`arm64`、`prod-builder`。
- 最大并发构建数。
- 启用/禁用状态。

#### 4.1.3 主机检测

新增或编辑主机时需要支持检测：

- SSH 连通性。
- Docker 命令可用性。
- Docker daemon 是否可访问。
- 当前用户是否有 Docker 权限。
- 主机架构检测，例如执行 `uname -m` 或 `docker info`。
- 可用磁盘空间。
- Docker 版本和 BuildKit 能力。

#### 4.1.4 主机状态展示

前端需要展示：

- 在线、离线、不可用、禁用、忙碌状态。
- 当前运行任务数和并发上限。
- 最近一次心跳或检测时间。
- 最近失败原因。
- 架构和标签。
- 磁盘空间和 Docker 版本。

### 4.2 镜像仓库管理

镜像仓库用于拉取基础镜像和推送构建产物。

#### 4.2.1 支持仓库类型

一期建议支持通用 Docker Registry 协议，基于该能力兼容：

- 自部署 Docker Registry。
- Harbor。
- Docker Hub。
- 阿里云容器镜像服务。
- 腾讯云容器镜像服务。
- 其他兼容 Docker Registry v2 的私有仓库。

后续可以增加云厂商专用适配器，用于自动获取命名空间、项目、地域等信息。

#### 4.2.2 仓库配置字段

- 仓库名称。
- 仓库类型。
- Registry 地址。
- 命名空间或项目。
- 地域，云厂商仓库可选。
- 用户名。
- 密码、访问令牌或临时凭据。
- 是否允许拉取基础镜像。
- 是否允许推送构建产物。
- 是否默认仓库。
- TLS 校验配置。
- 不安全 HTTP 仓库开关，仅限内网或测试环境使用。

#### 4.2.3 仓库操作

- 测试登录。
- 测试拉取指定基础镜像。
- 测试推送一个临时校验镜像，生产环境可以默认关闭。
- 查看仓库可用状态。
- 管理凭据更新。
- 对敏感信息进行加密存储和脱敏展示。

### 4.3 镜像项目、版本和分支管理

平台中会存在很多互不相同的镜像，例如 Java 镜像、Python 镜像、Node.js 镜像、MySQL 镜像等。平台中的“镜像项目”表示一条可持续演进的镜像构建线，例如 `company/base-java`、`company/python-runtime`。

版本和分支可视化不能把所有镜像混在一个全局图里展示。用户进入版本图前，需要先选择镜像项目或起始镜像，然后只查看该镜像线下的版本、分支、构建记录和产物。

#### 4.3.1 镜像目录和起始镜像选择

镜像项目管理页需要先提供镜像目录或镜像列表，作为版本图入口：

- 支持按镜像类型筛选，例如 Java、Python、Node.js、数据库、中间件、基础系统镜像。
- 支持按镜像名称、命名空间、维护人、标签和描述搜索。
- 每个镜像项目需要有明确的起始镜像，可以是外部基础镜像，如 `ubuntu:24.04`、`eclipse-temurin:17`，也可以是平台内某个已构建镜像产物。
- 点击某个镜像项目或起始镜像后，进入该镜像自己的版本和分支图。
- 版本图默认只展示当前选中镜像项目的主干、分支和版本节点。
- 如果某个镜像是从另一个镜像项目的版本节点衍生出来的，需要记录来源项目和来源节点，并可在详情中跳转到来源。
- 后续可支持跨镜像依赖视图，用于查看某个基础镜像被哪些下游镜像继承或引用。

#### 4.3.2 镜像项目

镜像项目字段建议包括：

- 项目名称。
- 镜像类型，如 Java、Python、Node.js、MySQL、基础系统镜像。
- 镜像名称。
- 所属命名空间。
- 起始镜像引用。
- 起始镜像来源：外部镜像、平台内镜像产物、其他镜像项目版本节点。
- 默认基础镜像。
- 默认目标仓库。
- 默认构建架构。
- 项目标签。
- 项目描述。
- 维护人。
- 创建时间和更新时间。

#### 4.3.3 版本节点

每次重要变更可以形成一个版本节点：

- 节点 ID。
- 版本号或标签，如 `v1.0.0`、`jdk17-202607`。
- 父节点 ID。
- 所属分支。
- Dockerfile 内容。
- 表单化构建配置快照。
- 变更说明。
- 构建参数。
- 创建人和创建时间。
- 最近一次成功构建产物。

#### 4.3.4 分支

从任意版本节点可以创建分支：

- 分支名称，如 `main`、`dev`、`arm64`、`java17`。
- 起始版本节点。
- 当前最新节点。
- 分支说明。
- 启用/归档状态。

#### 4.3.5 可视化展示

前端需要用图形方式展示版本和分支关系：

- 进入版本图前必须先选择镜像项目或起始镜像。
- 版本图标题和筛选条件需要明确显示当前选中的镜像项目、起始镜像和默认仓库。
- 支持展示主干和分支。
- 节点显示版本号、构建状态、创建时间。
- 点击节点后展示 Dockerfile、表单配置、说明、构建记录和镜像产物。
- 支持从节点创建新版本。
- 支持从节点创建分支。
- 支持比较两个版本的 Dockerfile 差异。
- 支持筛选分支、构建状态和创建人。

### 4.4 Dockerfile 和表单化构建

平台需要同时支持直接编辑 Dockerfile 和通过表单生成 Dockerfile。

#### 4.4.1 直接编辑 Dockerfile

- 提供 Dockerfile 编辑器。
- 支持语法高亮。
- 支持基础校验，例如是否存在 `FROM`。
- 支持保存草稿。
- 支持构建前预览。
- 支持查看历史版本。

#### 4.4.2 表单化构建

表单模式用于降低 Dockerfile 编写门槛。

建议表单字段包括：

- 基础镜像。
- 目标镜像名称和 tag。
- 构建架构：amd64、arm64 等。
- 系统包安装命令。
- 语言运行时或工具安装配置。
- 环境变量。
- 工作目录。
- 文件复制规则。
- 启动命令 `CMD`。
- 入口命令 `ENTRYPOINT`。
- 暴露端口。
- 用户和用户组。
- 构建参数 `ARG`。
- 构建密钥，必须避免写入最终镜像层。
- 标签 `LABEL`。

表单生成的 Dockerfile 需要允许用户切换到源码模式继续编辑。源码模式编辑后，系统应提示可能无法完整反向还原为表单。

#### 4.4.3 构建上下文

平台需要明确支持构建上下文来源：

- 仅 Dockerfile，无额外文件。
- 上传文件压缩包作为构建上下文。
- 从 Git 仓库拉取构建上下文，作为后续增强。
- 从已有模板复制构建上下文，作为后续增强。

构建上下文需要限制大小、记录来源并支持清理策略。

### 4.5 构建任务流程

核心构建流程如下：

1. 用户选择或创建镜像项目。
2. 用户确认该项目的起始镜像，并进入该镜像项目的版本图。
3. 用户选择版本节点或新建版本节点。
4. 用户直接编辑 Dockerfile，或通过表单生成 Dockerfile。
5. 用户选择构建架构、构建主机策略、目标镜像 tag 和目标仓库。
6. 后端创建构建任务，进入排队状态。
7. 调度器根据架构、主机标签、主机状态、并发限制选择构建主机。
8. 后端将 Dockerfile 和构建上下文发送到目标主机。
9. 目标主机执行 Docker 构建命令。
10. 后端实时采集构建日志并推送给前端。
11. 构建成功后记录镜像 ID、digest、tag、构建耗时等信息。
12. 按配置登录目标仓库并推送镜像。
13. 推送成功后记录仓库地址、镜像完整引用、digest 和推送时间。
14. 构建失败或推送失败时记录错误原因，并允许用户查看日志和重试。

#### 4.5.1 任务状态

构建任务至少包含以下状态：

- 草稿。
- 已提交。
- 排队中。
- 准备构建上下文。
- 分发中。
- 构建中。
- 构建成功。
- 构建失败。
- 推送中。
- 推送成功。
- 推送失败。
- 已取消。
- 已超时。

#### 4.5.2 构建日志

- 支持实时日志。
- 支持按任务查看完整日志。
- 支持日志搜索和下载。
- 支持区分系统日志、构建输出和推送输出。
- 支持失败日志定位，尽量提取最后错误片段。
- 支持日志保留策略。

#### 4.5.3 构建参数

构建任务应支持：

- `--build-arg`。
- `--platform`。
- `--no-cache`。
- `--pull`。
- 构建超时时间。
- 构建标签。
- Dockerfile 路径。
- 构建上下文路径。
- 目标 tag。
- 是否自动推送。

### 4.6 镜像产物管理

这里管理的是已构建完成的镜像产物，而不是运行中的容器。

需要展示：

- 镜像名称。
- tag。
- digest。
- 镜像 ID。
- 架构。
- 大小。
- 来源项目、版本节点和构建任务。
- 构建主机。
- 目标仓库。
- 推送状态。
- 创建时间。
- 创建人。

需要支持操作：

- 查看详情。
- 查看关联 Dockerfile。
- 查看构建日志。
- 复制镜像拉取命令。
- 重新构建。
- 重新推送。
- 删除本地构建产物，需按权限控制。
- 标记废弃或归档。

### 4.7 后台操作引导

后台管理系统需要内置操作指导，帮助用户完成常见配置。

#### 4.7.1 添加构建主机引导

引导内容包括：

- 如何确认目标主机已安装 Docker。
- 如何确认当前用户有 Docker 权限。
- 如何查看主机架构。
- 如何准备 SSH 密钥。
- 如何测试 SSH 连接。
- 如何配置最大并发数。
- 如何理解容器部署时的 Docker socket 挂载风险。

#### 4.7.2 查看主机架构引导

提供命令示例：

```bash
uname -m
docker info --format '{{.Architecture}}'
docker version
```

常见映射说明：

- `x86_64` 通常对应 `amd64`。
- `aarch64` 通常对应 `arm64`。
- `armv7l` 通常对应 `arm/v7`。

#### 4.7.3 镜像仓库配置引导

引导内容包括：

- 如何填写 Registry 地址。
- 如何区分命名空间、项目和镜像名称。
- 如何创建访问令牌。
- 如何测试登录。
- 如何配置不安全 HTTP 仓库。
- 如何选择默认拉取仓库和默认推送仓库。

#### 4.7.4 构建失败排查引导

引导内容包括：

- 基础镜像拉取失败。
- Dockerfile 语法错误。
- 构建主机架构不匹配。
- 构建上下文缺少文件。
- 网络代理或 DNS 问题。
- 磁盘空间不足。
- 仓库登录失败。
- 推送权限不足。

## 6. 权限和安全需求

### 6.1 认证和权限

平台需要支持：

- 用户登录。
- 角色权限控制。
- 管理员、维护者、查看者三类基础角色。
- 按镜像项目授权，作为后续增强。
- API token，作为后续增强。

### 6.2 凭据安全

- SSH 密钥、仓库密码、访问令牌必须加密存储。
- 前端不展示明文凭据。
- 凭据更新时允许覆盖，但不允许回显原值。
- 构建日志中需要尽量脱敏敏感值。
- 构建密钥不能直接写入 Dockerfile 的最终镜像层。

### 6.3 构建安全

- Dockerfile 具有执行任意命令的能力，应按权限限制谁可以发起构建。
- 容器部署并挂载 Docker socket 时，平台容器基本具备宿主机 Docker 控制能力，需要在文档和界面中明确提示。
- 远程主机 SSH 凭据应最小权限化，推荐使用专用构建用户。
- 支持构建超时，避免任务无限运行。
- 支持并发限制，避免构建主机资源被打满。

## 7. 部署需求

### 7.1 二进制部署

需要提供：

- 单个后端可执行文件。
- 前端静态资源可内嵌或随包分发。
- 配置文件示例。
- 数据目录配置。
- 日志目录配置。
- 数据库配置，初期可以支持 SQLite，后续支持 PostgreSQL 或 MySQL。
- 启动、停止、升级和备份说明。

### 7.2 Docker 部署

需要提供：

- Docker 镜像。
- docker run 示例。
- docker compose 示例。
- 挂载配置目录、数据目录和日志目录。
- 可选挂载 `/var/run/docker.sock` 以访问宿主机 Docker。
- 配置环境变量。
- 健康检查。

Docker 部署示例需要明确：

- 不挂载 Docker socket 时，只能使用远程 SSH、Docker API 或 Agent 主机构建。
- 挂载 Docker socket 后，平台可以使用宿主机 Docker 作为本机构建主机。
- 生产环境应优先使用远程构建主机或 Agent，降低平台服务容器直接控制宿主机的风险。

## 8. 可观测性和运维需求

- 后端服务需要输出结构化日志。
- 构建任务日志需要单独保存。
- 提供健康检查接口。
- 提供主机检测接口。
- 提供仓库检测接口。
- 记录审计日志：用户登录、创建主机、修改仓库、发起构建、删除产物等。
- 支持日志保留天数配置。
- 支持构建上下文和临时文件清理。
- 后续可支持 Prometheus 指标。

## 9. 通知需求

一期可先在后台展示任务状态。后续建议支持：

- 构建成功通知。
- 构建失败通知。
- 主机离线通知。
- 仓库登录失败通知。
- Webhook。
- 邮件。
- 企业微信、钉钉、飞书等机器人。

## 10. 推荐的一期范围

一期目标是跑通完整闭环：

- 前端管理界面。
- 用户登录和管理员账号初始化。
- 本机构建主机。
- SSH 远程构建主机。
- 通用 Docker Registry 管理。
- 镜像项目、版本节点和分支基础管理。
- Dockerfile 编辑构建。
- 简单表单生成 Dockerfile。
- 构建任务队列。
- 实时构建日志。
- 构建成功后推送到指定仓库。
- 镜像产物列表。
- 基础操作引导。
- 二进制部署和 Docker 部署。

## 12. 待确认问题

后续进入架构设计和实现前，需要确认：

1. 后端技术栈使用什么语言和框架。
2. 前端技术栈使用什么框架。
3. 一期数据库是否采用 SQLite，还是直接使用 PostgreSQL 或 MySQL。
4. 是否必须支持离线内网环境。
5. 是否需要接入公司已有用户体系，例如 LDAP、OIDC、企业微信或钉钉。
6. 是否需要支持 Windows 构建主机，还是一期仅支持 Linux。
7. 构建上下文是否一期就需要支持 Git 仓库。
8. 镜像漏洞扫描是否作为一期必要功能。
9. 是否需要审批流程，例如高权限镜像或生产仓库推送前审批。
10. 是否需要支持集群化部署和多个后端实例。
