# MVP 范围和验收标准

## 1. MVP 目标

MVP 的目标是交付一个可以实际运行的私有化镜像构建平台，完成从“配置构建环境”到“构建并推送镜像”的完整闭环。

MVP 必须证明以下能力成立：

- 平台可以通过二进制或 Docker 方式部署。
- 管理员可以配置本机构建和 SSH 远程构建主机。
- 管理员可以配置兼容 Docker Registry v2 的镜像仓库。
- 镜像维护者可以创建镜像项目，选择起始镜像，并维护版本和分支。
- 用户可以直接编写 Dockerfile 或通过简单表单生成 Dockerfile。
- 用户可以发起构建任务，实时查看日志。
- 构建成功后，平台可以将镜像推送到指定仓库。
- 平台可以记录构建产物、构建日志、失败原因和基础审计信息。

MVP 不追求大规模分布式调度，不追求完整 CI/CD，不追求所有云厂商深度适配。第一版重点是把核心构建链路做稳定。

## 2. 范围原则

MVP 范围按以下原则控制：

- 优先完整闭环，不优先高级扩展。
- 优先清晰状态，不优先自动化魔法。
- 优先可观察和可排错，不优先隐藏细节。
- 优先 Linux 和 Docker，暂不支持 Windows 构建主机。
- 优先单实例部署，暂不支持多后端实例。
- 优先 Docker CLI 行为一致性，暂不直接暴露复杂 Docker Engine API 能力。

## 3. 用户角色

MVP 包含三类角色：

| 角色 | MVP 能力 |
| --- | --- |
| admin | 初始化系统、管理用户、管理主机、管理仓库、查看所有任务和产物 |
| maintainer | 管理镜像项目、版本、分支、Dockerfile，发起构建 |
| viewer | 查看项目、版本图、构建记录、日志和产物 |

MVP 权限不做项目级细粒度授权。项目级授权作为后续增强。

## 4. MVP 功能清单

### 4.1 系统初始化和登录

必须支持：

- 首次启动时创建管理员账号。
- 管理员登录后台。
- 登录态通过 HttpOnly Cookie 保存。
- 用户退出登录。
- 查看当前登录用户信息。
- 基础角色控制。

暂不支持：

- LDAP。
- OIDC。
- SSO。
- API token。
- 多租户。

### 4.2 构建主机管理

必须支持：

- 默认本机构建主机。
- 添加 SSH 远程构建主机。
- 编辑主机基本信息。
- 启用、禁用主机。
- 删除未被任务引用的主机。
- 检测主机连接状态。
- 检测 Docker 是否可用。
- 检测主机架构。
- 检测 Docker 版本。
- 配置主机最大并发数。
- 配置主机标签。

本机构建验收：

- 二进制部署时，可以调用当前机器 Docker 完成构建。
- Docker 部署并挂载 `/var/run/docker.sock` 时，可以调用宿主机 Docker 完成构建。
- Docker socket 未挂载时，界面明确提示本机构建不可用。

SSH 主机验收：

- 可以通过 SSH 密钥连接远程 Linux 主机。
- 可以上传构建上下文到远程临时目录。
- 可以在远程主机执行 `docker build`。
- 任务结束后清理远程临时目录。

暂不支持：

- Agent 主机。
- Kubernetes 节点。
- Docker API TCP 主机。
- Windows 主机。

### 4.3 镜像仓库管理

必须支持：

- 添加通用 Docker Registry v2 仓库。
- 支持 Harbor、自部署 Registry、Docker Hub、阿里云、腾讯云等兼容仓库的基础连接方式。
- 配置 Registry 地址、命名空间、用户名、密码或 token。
- 设置仓库是否可用于拉取基础镜像。
- 设置仓库是否可用于推送产物。
- 设置默认推送仓库。
- 测试登录。
- 凭据加密存储。
- 前端脱敏展示凭据。

验收标准：

- 使用正确凭据时，仓库检测成功。
- 使用错误凭据时，仓库检测失败并显示可理解的错误。
- 构建推送时，平台使用临时 Docker 配置登录仓库。
- 构建日志中不出现仓库密码或 token 明文。

暂不支持：

- 云厂商 API 深度集成。
- 自动拉取命名空间列表。
- 镜像仓库内镜像浏览。

### 4.4 镜像项目和起始镜像

必须支持：

- 创建镜像项目。
- 设置镜像类型，例如 Java、Python、Node.js、MySQL、Base OS。
- 设置镜像名称、命名空间、默认仓库、默认架构。
- 设置起始镜像引用。
- 起始镜像可以是外部镜像引用，例如 `ubuntu:24.04`。
- 展示镜像项目列表。
- 按名称、类型、标签搜索和筛选。
- 进入某个镜像项目后，只查看该项目的版本和分支图。

验收标准：

- Java、Python、Node.js、MySQL 等不同镜像项目不会混在同一个版本图中。
- 用户必须先选择镜像项目或起始镜像，再进入版本图。
- 版本图标题明确显示当前镜像项目和起始镜像。

暂不支持：

- 跨镜像依赖全局视图。
- 从平台内其他镜像产物直接创建起始镜像的完整向导。

### 4.5 版本和分支管理

必须支持：

- 为镜像项目创建版本节点。
- 从任意版本节点创建子版本。
- 从任意版本节点创建分支。
- 查看分支列表。
- 归档分支。
- 查看版本节点详情。
- 查看版本节点 Dockerfile。
- 查看版本节点说明。

版本图必须采用 Git-style 节点连线形式：

- 节点代表镜像版本。
- 连线代表父子版本关系。
- 图上只展示版本号、分支、状态和时间等少量信息。
- 点击节点后，在详情面板查看完整信息。

验收标准：

- 主干和分支关系清晰可见。
- 点击节点能打开详情。
- 从节点创建新版本后，图中出现新节点和连线。
- 从节点创建分支后，图中出现新分支线。

暂不支持：

- 复杂合并策略。
- 自动语义化版本推荐。
- 跨项目版本图。

### 4.6 Dockerfile 编辑和表单构建

必须支持：

- Dockerfile 源码编辑。
- Dockerfile 只读查看。
- 基础语法校验，例如必须存在 `FROM`。
- 保存 Dockerfile 到版本节点。
- 通过简单表单生成 Dockerfile。
- 表单生成后允许切换到源码模式继续编辑。

MVP 表单字段：

- 基础镜像。
- 目标镜像 tag。
- 构建架构。
- 环境变量。
- 工作目录。
- 系统包安装命令。
- 文件复制规则。
- 暴露端口。
- `CMD`。
- `ENTRYPOINT`。
- `ARG`。
- `LABEL`。

验收标准：

- 表单填写后能生成可预览的 Dockerfile。
- 直接编辑 Dockerfile 后可以保存并用于构建。
- 无 `FROM` 的 Dockerfile 不能提交构建。

暂不支持：

- 完整 Dockerfile AST 解析。
- 源码模式反向完整还原表单。
- BuildKit secret 表单化配置。

### 4.7 构建任务

必须支持：

- 从版本节点发起构建。
- 选择构建主机或按架构自动选择主机。
- 选择目标仓库。
- 设置目标镜像 tag。
- 设置 `--build-arg`。
- 设置 `--no-cache`。
- 设置 `--pull`。
- 设置构建超时。
- 提交任务后进入队列。
- 查看任务状态。
- 取消排队中或构建中的任务。
- 失败后重试。

任务状态必须覆盖：

- `created`
- `queued`
- `preparing_context`
- `dispatching`
- `building`
- `build_success`
- `pushing`
- `push_success`
- `preparing_context_failed`
- `dispatch_failed`
- `build_failed`
- `push_failed`
- `canceled`
- `timeout`

验收标准：

- 构建任务状态能准确流转。
- 构建失败时能看到失败阶段和错误摘要。
- 推送失败不会被误标为构建失败。
- 重试任务会保留和原任务的关联关系。

### 4.8 实时日志

必须支持：

- SSE 实时推送构建日志。
- 任务详情页实时展示日志。
- 构建完成后查看完整历史日志。
- 下载日志文件。
- 日志中区分系统阶段、构建输出、推送输出。
- 日志中尽量脱敏敏感值。

验收标准：

- 构建过程中前端无需刷新即可看到日志。
- SSE 断开后前端可以自动重连。
- 长日志不会导致页面明显卡顿。
- 构建结束后日志仍可查看。

### 4.9 镜像产物管理

必须支持：

- 构建成功后创建镜像产物记录。
- 记录镜像名称、tag、digest、架构、大小、来源任务、来源版本节点、目标仓库。
- 查看产物详情。
- 复制 `docker pull` 命令。
- 重新推送产物。
- 标记产物归档或废弃。

验收标准：

- 推送成功后产物显示完整镜像引用。
- 用户可以从产物回到构建任务、版本节点和项目。
- 重新推送失败时不影响原产物记录。

暂不支持：

- 镜像漏洞扫描。
- SBOM。
- 自动清理远端仓库镜像。

### 4.10 帮助和引导

必须支持内置帮助页面：

- 如何添加本机构建主机。
- 如何在 Docker 部署中挂载 Docker socket。
- Docker socket 风险说明。
- 如何添加 SSH 构建主机。
- 如何查看主机架构。
- 如何配置镜像仓库。
- 常见构建失败原因。

验收标准：

- 用户可以在后台找到添加主机、查看架构、配置仓库的操作说明。
- 高风险配置必须有明确提示，例如 Docker socket 挂载。

### 4.11 审计和设置

MVP 必须记录基础审计：

- 登录。
- 创建、修改、删除构建主机。
- 创建、修改、删除镜像仓库。
- 创建镜像项目。
- 创建版本节点和分支。
- 发起构建。
- 取消构建。
- 重新推送产物。

系统设置必须支持：

- 全局最大并发构建数。
- 默认构建超时。
- 日志保留天数。
- 构建上下文保留策略。

## 5. MVP 不包含内容

以下能力明确不进入 MVP：

- 多租户。
- 项目级权限。
- Agent。
- Kubernetes 调度。
- Docker API TCP 构建主机。
- Windows 构建主机。
- buildx 多架构 manifest。
- Git 仓库构建上下文。
- Webhook 触发构建。
- 定时构建。
- 漏洞扫描。
- SBOM。
- 镜像仓库浏览。
- 高可用部署。
- 多后端实例。
- Redis/NATS/Kafka 等外部队列。
- LDAP/OIDC/SSO。
- API token。

## 6. 质量门槛

### 6.1 后端质量

MVP 后端必须满足：

- 核心 service 有单元测试。
- 构建任务状态机有单元测试。
- Docker 命令拼装逻辑有单元测试。
- 凭据加密和脱敏有单元测试。
- API 返回统一错误格式。
- 所有外部命令执行必须支持 context 取消和超时。
- 所有构建临时目录必须有清理逻辑。

### 6.2 前端质量

MVP 前端必须满足：

- 关键页面可正常加载。
- 表单字段有基础校验。
- 列表页支持 loading、empty、error 状态。
- 构建日志长列表不卡顿。
- 版本图节点点击和详情面板稳定。
- 登录失效后能回到登录页。

### 6.3 发布质量

MVP 发布必须满足：

- 可以构建后端二进制。
- 可以构建前端静态资源。
- 可以构建 Docker 镜像。
- 提供 docker compose 示例。
- 提供默认配置文件示例。
- 提供初始化管理员账号方式。
- 提供升级前备份说明。

## 7. 验收场景

### 7.1 本机构建闭环

1. 使用二进制启动平台。
2. 初始化管理员账号。
3. 登录后台。
4. 检测默认本机构建主机。
5. 添加一个本地或私有 Registry。
6. 创建 Java 镜像项目。
7. 设置起始镜像 `eclipse-temurin:17`。
8. 创建版本节点并写入 Dockerfile。
9. 发起构建。
10. 实时查看日志。
11. 构建成功后推送镜像。
12. 在镜像产物列表看到结果。

### 7.2 Docker 部署闭环

1. 使用 Docker 启动平台并挂载 `/var/run/docker.sock`。
2. 登录后台。
3. 检测本机构建主机。
4. 创建 Python 镜像项目。
5. 发起构建并推送。
6. 确认产物可通过 `docker pull` 拉取。

### 7.3 SSH 远程构建闭环

1. 添加 SSH 构建主机。
2. 检测 SSH 和 Docker 能力。
3. 创建 Node.js 镜像项目。
4. 选择 SSH 主机发起构建。
5. 查看实时日志。
6. 构建成功后推送到仓库。
7. 确认远程临时目录被清理。

### 7.4 构建失败排查

1. 提交一个错误 Dockerfile。
2. 构建进入失败状态。
3. 任务详情显示失败阶段和错误摘要。
4. 日志中能定位 Dockerfile 错误。
5. 修正 Dockerfile 后重试成功。

### 7.5 推送失败排查

1. 使用错误仓库凭据。
2. 构建成功但推送失败。
3. 任务状态为 `push_failed`。
4. 修正仓库凭据。
5. 重新推送成功。

## 8. 完成定义

MVP 完成必须同时满足：

- `docs/requirements.md` 中一期能力已实现或明确延期。
- 本文档的验收场景全部可通过。
- 二进制部署和 Docker 部署均可运行。
- 至少一个本机构建和一个 SSH 远程构建场景通过。
- 构建日志、任务状态、产物记录可追踪。
- README 中有可执行的启动说明。
- Docker socket 风险、凭据安全和构建主机权限风险已在界面和文档中提示。
